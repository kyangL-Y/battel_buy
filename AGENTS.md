# 代码生成与编写补充规则

生成或修改代码时，除遵循项目既有规范外，还应满足以下要求：

## 1. 可读与可审阅

- 优先保证结构清晰、命名表意、逻辑分段合理，便于开发人员阅读与 Code Review。
- 避免为炫技而过度抽象，避免把过多逻辑压缩到单行或少量难审阅的表达式中。

## 2. 关键处注释

- 仅在业务规则、非显而易见的边界条件、与外部约定相关的决策处补充简短注释。
- 注释重点说明“为什么这样做”，不写显而易见的废话注释。
- 外部约定包括但不限于协议、表结构、权限约束、上下游接口契约。

## 3. 避免无谓拆方法

- 若某段逻辑在功能上不存在不少于 2 处调用或复用价值，不刻意拆成细碎私有方法。
- 优先在同一作用域内通过清晰分段、局部变量和必要注释完成表达，降低阅读跳转成本。
- 仅当存在明确复用、对外导出 API 复用场景，或拆分后能显著提升可读性与边界表达时，才提炼方法或抽象。
